## How to Read This Backlog

Each item has a short title, description, category, expected benefit, and four assessment columns left blank for prioritization review:

| Column | Meaning |
|--------|---------|
| **Zero-dep** | Can this feature be implemented without adding new runtime dependencies to the project? A checkmark means it builds entirely on what we already ship (tree-sitter, SQLite, existing AST). Blank means it needs evaluation. Features that require new deps raise the install footprint and maintenance burden ‚Äî they need stronger justification. |
| **Foundation-aligned** | Does this feature align with the [FOUNDATION.md](../../FOUNDATION.md) core principles? Specifically: does it keep the graph always-current (P1), maintain zero-cost core with optional LLM enhancement (P4), respect embeddable-first design (P5), and optimize for planning, developing, and refactoring ‚Äî not linting or CI/CD (P8)? A checkmark means full alignment. An X means it conflicts with at least one principle and needs a deliberate exception. |
| **Problem-fit (1-5)** | How directly does this feature address the core problem from our README: *AI coding assistants waste tokens re-orienting themselves in large codebases, hallucinate dependencies, and miss blast radius.* A 5 means it directly reduces token waste, prevents hallucinated deps, or catches breakage. A 1 means it's tangential ‚Äî nice to have but doesn't solve the stated problem. |
| **Breaking** | Is this a breaking change? `Yes` means existing CLI output, API signatures, DB schema, or MCP tool contracts change in incompatible ways. `No` means it's purely additive. Breaking changes require a major version bump. |
| **Depends on** | List of backlog IDs that must be completed before this item can be implemented. `‚Äî` means no dependencies. When an item is completed, scan this column to find what it unlocks ‚Äî those items are now unblocked and should be prioritized. |

## Scoring Guide

When filling in the assessment columns during a prioritization session:

**Zero-dep checklist:**
- Does it use only tree-sitter AST data we already extract? ‚Üí likely zero-dep
- Does it need a new npm package at runtime? ‚Üí not zero-dep
- Does it need git CLI access? ‚Üí acceptable (git is already assumed)
- Does it need a new WASM module or native addon? ‚Üí not zero-dep

**Foundation alignment red flags:**
- Adds a cloud API call to the core pipeline ‚Üí violates P1 and P4
- Requires Docker, external DB, or non-npm toolchain ‚Üí violates zero-infrastructure goal
- Generates code, edits files, or makes decisions ‚Üí violates P8 (we're not an agent)
- Primarily optimizes for CI/CD or linting use cases ‚Üí deprioritized by P8 (we optimize for planning, developing, and refactoring)
- Breaks programmatic API contract ‚Üí check against P5 (embeddable-first)

**Problem-fit rubric:**
- **5** ‚Äî Directly reduces token waste, prevents hallucinated dependencies, or catches blast-radius breakage
- **4** ‚Äî Improves agent accuracy or reduces round-trips for common tasks
- **3** ‚Äî Useful for developers and agents but doesn't address the core "lost AI" problem
- **2** ‚Äî Nice-to-have; improves the tool but tangential to the stated problem
- **1** ‚Äî Cool feature, but doesn't help AI agents navigate codebases better
